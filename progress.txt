PROJECT OVERVIEW
--------------------------------------------------------------------------------
Sari-Sari Smart is a mobile-first daily business companion for traditional sari-sari
store owners. Refactored from an item-by-item POS system to a daily awareness tool
based on interview findings. Three pillars: Stock Tracker | Daily Cash Summary |
Utang Notebook.

ARCHITECTURE
================================================================================

Multi-Page Structure:
  - 11 HTML files, each with its own phone frame
  - Shared style.css for consistent theming
  - Shared app.js with page-aware initialization (switch/case per page name)

Navigation:
  - Bottom nav uses <a> tags for native page navigation
  - 5 main nav pages (home, stocks/inventory, sales, utang/debts, help) have bottom nav
  - Sub-pages (new_sale, add_product, new_debt, setting) have NO bottom nav
  - Sub-pages highlight parent nav tab via pageToNav mapping
  - Settings gear passes ?from=PAGENAME for smart back-button destination

State Management:
  - In-memory state object with full application data
  - localStorage keys: sss_settings, sss_products, sss_sales, sss_customers,
    sss_debts, sss_usedNames, sss_stockStatuses, sss_dailyEntries,
    sss_specificSales, sss_eodData
  - New state fields: stockStatuses, dailyEntries, specificSales, endOfDayData

Localization:
  - Two languages: English (default) and Filipino
  - data-i18n attributes on all user-facing text
  - applyLanguage() updates all text dynamically + refreshes all views
  - Translation dictionary with 100+ keys per language

Data Flow (v2.2):
  1. User visits home → cards show real-time summaries of all data
  2. Sales page → enter gastos + kita → dailyEntry saved → home card updates
  3. Stocks page → tap to cycle status, Sold button to deduct → stockStatuses updated
  4. Specific sale → enter product name (autocomplete from inventory) + quantity
     → total auto-calculates as sellingPrice × qty → inventory directly deducted
  5. Specific sale → amount reflected in Today's Earnings on home (fallback if no daily entry)
  6. Specific sale → auto-creates debt if customer named
  7. End-of-Day → 3 steps → eodData saved → home card shows completion
  8. Settings → saved to state → applied immediately

Tutorial:

  3. Choose payment details:
     - Cash
     - GCash
     - Split payment
     - Debt/Utang

The user could speak naturally, for example:
  "Two pieces of bread worth 20 pesos, payment via GCash, split in half,
   the other half is cash."

The system would:
  1. Convert speech to text
  2. Use AI/NLP to analyze the intent
  3. Extract: product name, quantity, amount, payment method,
     debt information (if any)
  4. Automatically record the transaction
