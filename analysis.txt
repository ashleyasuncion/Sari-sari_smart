  [ ] Testing: Unit tests (JUnit) + UI tests (Espresso or Compose UI tests)
  [ ] All 3 pillars ported from this prototype:

        - Daily Cash Summary (Gastos + Kita → auto profit/revolving fund)

===============================================================================
>> CRITICAL CONCERNS
===============================================================================

=== 1. This is NOT a capstone-ready deliverable. It is a prototype. ===

Your own analysis notes that this is a "rapid design/UX prototype for the actual
capstone deliverable, which will be built in Android Studio." The problem is
clear: there is no Android app visible. If you walk into a capstone defense with
only HTML/CSS/JS, you will fail — or at best be asked to resubmit.

A capstone in Android development must demonstrate:
  - MVVM architecture (ViewModel + LiveData/StateFlow)
  - Room database (local SQLite persistence)
  - Material Design 3 (Jetpack Compose or XML)
  - Unit tests (JUnit)
  - UI tests (Espresso or Compose UI tests)

=== 2. Where is the user research evidence? ===

The instructions reference interviews with sari-sari store owners and a
transcript, but none of these artifacts are in the repository. In a capstone
defense, the first question a panelist asks is: "Show me your research process."
If the answer is just "we talked to people," that is not sufficient.

At minimum, the repository should include:
  - 3-5 interview summaries (anonymized)
  - A before/after comparison showing how interview insights changed the design
  - At least one usability test result showing measurable improvement

=== 3. localStorage is NOT a database ===

The entire app runs on localStorage. This is acceptable for a prototype, but
as a capstone deliverable, you must demonstrate understanding of:
  - Why localStorage has size limits (~5-10MB)
  - Why localStorage is synchronous and blocks the UI thread
  - Why localStorage does not support queries, indexes, or transactions
  - How Room/SQLite solves all of these problems

Expect the question: "How would you migrate this from localStorage to Room?"
You need a concrete answer, not a vague plan.

=== 4. The code has structural issues ===

The app.js file is a ~2,000+ line monolithic IIFE with no modularity. This
is fine for a prototype sprint, but does not demonstrate software engineering
best practices. At a capstone defense, expect questions like:
  - "How would you test this code?"
  - "How would you separate concerns?"
  - "What design patterns did you use?"

For the Android version, you need proper architecture (MVVM, Repository
pattern, Dependency Injection).
